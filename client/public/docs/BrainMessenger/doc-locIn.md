# Руководство по Локализации BrainMessenger (Доступность и Масштабирование)

### 1. Введение: Расширение Ценности через Доступность

**Название проекта:** BrainMessenger

**Описание:** Это руководство описывает процесс локализации BrainMessenger — ключевой шаг для обеспечения **доступности** и **ценности** (Принцип 2) продукта для широкой аудитории по всему миру. Мы рассматриваем локализацию не просто как перевод строк, а как **систематический процесс** (Принцип 9) адаптации интерфейса и контента, который позволит нам **масштабировать наше влияние** (Принцип 2, 8).

**Цель:** Сделать BrainMessenger понятным и удобным для пользователей из разных культур, начиная с базового набора языков и создавая **надежную систему** (Принцип 9) для легкого добавления новых в будущем.

**Текущий статус:** Базовая поддержка английского языка (MVP). Идет планирование и подготовка к добавлению новых языков в рамках текущих этапов разработки (см. Дорожную карту).

**Принципы, которыми руководствуется локализация:**

*   **Создание Ценности (Принцип 2):** Локализация напрямую повышает ценность продукта для неанглоязычных пользователей.
*   **Система и Оптимизация (Принцип 9):** Процесс локализации строится как четкая, повторяемая система для эффективности.
*   **Прагматизм и Реализм (Принцип 12):** Выбор начального набора языков основывается на потенциальном охвате и ресурсах.
*   **Внешняя Экспертиза и Обратная Связь (Принцип 14):** Привлечение профессиональных переводчиков и сбор обратной связи от пользователей на разных языках.

### 2. Цели Локализации

*   Обеспечить поддержку **минимум 5 ключевых языков** (Английский, Испанский, Французский, Русский, Арабский) для старта MVP — (Требование NFR-11, соответствующее Принципу 12: Прагматизм).
*   Реализовать **адаптацию пользовательского интерфейса для языков с направлением письма справа налево (RTL)**, таких как арабский (Принцип 2: Доступность).
*   Поддерживать **единообразие терминологии и стиля** во всех переводах, используя централизованные инструменты (Принцип 9: Система).
*   Создать **эффективный и простой процесс** добавления новых языков по мере роста базы пользователей и поступления запросов (Принцип 9: Масштабирование, Принцип 12: Прагматизм, Принцип 14: Обратная связь).

### 3. Инструменты и Технологии: Надежный Фундамент (Принцип 8, 9)

Выбор инструментов основан на **прагматизме** (Принцип 12), **надежности** и поддержке **системного подхода** (Принцип 9).

### 4. Структура Файлов Переводов: Порядок в Системе (Принцип 9)

Переводы организованы в четкой файловой структуре для удобства управления (Принцип 9: Система).

#### 4.1. Расположение

Файлы переводов хранятся в директории `locales` внутри `frontend/src`:

frontend/
├── src/
│ ├── locales/
│ │ ├── en.json # Английский (базовый)
│ │ ├── es.json # Испанский
│ │ ├── fr.json # Французский
│ │ ├── ru.json # Русский
│ │ ├── ar.json # Арабский (пример RTL)
│ └── index.ts # Конфигурация i18next

#### 4.2. Формат JSON: Контекст и Гибкость

*   **Ключи:** Используется `camelCase`. Ключи должны быть **описательными и отражать контекст** использования строки (например, `sendButton` вместо просто `send`).
*   **Значения:** Содержат переводимый текст. Поддерживается **интерполяция** для вставки динамических данных (например, имена пользователей).
*   **Пример `en.json`:**

```json
{
  "welcomeTitle": "Welcome to BrainMessenger!",
  "sendButton": "Send",
  "errors": {
    "unauthorized": "Please log in again to continue.",
    "notFound": "The requested item could not be found."
  },
  "chat": {
    "newMessageNotification": "New message from {{name}}",
    "unreadCount": "{{count}} unread messages"
  },
  "settings": {
    "language": "Language",
    "theme": "Theme"
  }
}

4.3. RTL-флаг: Адаптация Интерфейса
Для языков с направлением справа налево в начало соответствующего JSON-файла добавляется метаданные для индикации:

{
  "rtl": true,
  "welcomeTitle": "مرحبًا بك في BrainMessenger!"
  // ... остальные переводы
}

Этот флаг используется Frontend-приложением для соответствующей адаптации стилей и макета.

5. Настройка Локализации: Интеграция в Систему (Принцип 9)
Локализация интегрируется как в Frontend (для отображения переведенного UI), так и в Backend (для возврата локализованных ошибок или уведомлений).
5.1. Frontend (React Native): Использование i18next
1. Инициализация i18next:
В главном файле приложения или файла конфигурации (frontend/src/index.ts или подобном):

import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'

// Импорт всех файлов локалей
import en from './locales/en.json'
import es from './locales/es.json'
import fr from './locales/fr.json'
import ru from './locales/ru.json'
import ar from './locales/ar.json'

// Определение ресурсов
const resources = {
  en: { translation: en },
  es: { translation: es },
  fr: { translation: fr },
  ru: { translation: ru },
  ar: { translation: ar },
};

i18n
  .use(initReactI18next) // Подключение react-i18next
  .init({
    resources,
    lng: 'en', // Язык по умолчанию при первом запуске (может определяться по системным настройкам или выбору пользователя)
    fallbackLng: 'en', // Резервный язык, если текущий перевод отсутствует
    interpolation: {
      escapeValue: false // Позволяет использовать HTML теги в переводах (осторожно!)
    }
  });

export default i18n;

2. Использование перевода в компонентах:
Используйте хук useTranslation для доступа к функции t и объекту i18n.

import { useTranslation } from 'react-i18next';
import { Text, Button, View } from 'react-native';

const WelcomeScreen = () => {
  const { t, i18n } = useTranslation(); // Получаем функцию t и объект i18n

  return (
    <View>
      {/* Использование простого перевода */}
      <Text>{t('welcomeTitle')}</Text>
      
      {/* Использование перевода с интерполяцией */}
      {/* Предполагается, что где-то есть состояние unreadCount */}
      <Text>{t('chat.unreadCount', { count: 5 })}</Text> 

      {/* Использование перевода для текста кнопки */}
      <Button title={t('sendButton')} onPress={sendMessage} />
      
      {/* Пример смены языка */}
      <Button title={t('settings.language')} onPress={() => i18n.changeLanguage('ar')} />
    </View>
  );
};

3. Смена языка:
Язык меняется вызовом i18n.changeLanguage('код_языка'). Обычно это делается в настройках пользователя.

5.2. Backend (NestJS): Локализация Ошибок и Уведомлений
Backend может возвращать локализованные сообщения об ошибках или уведомления, опираясь на язык, переданный Frontend (например, в заголовке Accept-Language или настройках пользователя).

Использование локализованных сообщений при выбрасывании ошибок:

import { HttpException, HttpStatus } from '@nestjs/common';
import { t } from 'i18next'; // Предполагается, что i18next инициализирован и в Backend

// В сервисе или контроллере
if (!user) {
  // Определяем язык пользователя (например, из request.headers['accept-language'])
  const userLang = determineUserLanguage(request); 
  throw new HttpException(t('errors.unauthorized', { lng: userLang }), HttpStatus.UNAUTHORIZED);
}

Локаль пользователя определяется из заголовка Accept-Language (стандартный подход) или сохраняется в профиле пользователя в базе данных после его выбора в настройках приложения.

5.3. RTL-адаптация (Справа Налево): Зеркальное Отображение Системы
Для языков с RTL (арабский, иврит и др.) необходимо зеркально отобразить макет и некоторые элементы интерфейса.

Стили (CSS/CSS-in-JS):
Используйте rtlcss в процессе сборки или условные стили в коде. rtlcss автоматически преобразует свойства вроде margin-left, padding-right, text-align: left и т.п. в их RTL-эквиваленты (margin-right, padding-left, text-align: right).

/* Исходный CSS для LTR */
.message-bubble {
  margin-left: 10px;
  text-align: left;
}
/* После обработки rtlcss для RTL */
.message-bubble {
  margin-right: 10px;
  text-align: right;
}

React Native:
React-Native имеет встроенную поддержку RTL через I18nManager.

import { I18nManager } from 'react-native';
import i18n from './index'; // Ваш файл инициализации i18next

// В главном файле приложения, перед рендером
const isRTL = i18n.language === 'ar' || i18n.language === 'he'; // Проверяем, является ли текущий язык RTL

// ForceRTL может потребовать перезагрузки приложения для полной силы
if (isRTL !== I18nManager.isRTL) {
  I18nManager.forceRTL(isRTL);
  // Optional: Restart the app to apply RTL layout fully
  // RNRestart.Restart(); 
}

// Некоторые стили могут требовать ручной адаптации, если auto-conversion недостаточно
const containerStyle = {
  flexDirection: isRTL ? 'row-reverse' : 'row',
  textAlign: isRTL ? 'right' : 'left',
};

Иконки: Некоторые иконки (например, стрелки "назад", "далее") должны быть зеркально отображены для RTL. Это можно делать условно в коде или использовать RTL-ready наборы иконок.

6. Процесс Добавления Нового Языка: Масштабируемая Система (Принцип 9)
Процесс добавления нового языка построен как четкая, повторяемая последовательность действий для обеспечения эффективности (Принцип 9).
Создание базового файла перевода:
Скопируйте актуальный en.json (или другой наиболее полный файл) в frontend/src/locales/<code>.json, где <code> — двухбуквенный код нового языка (например, de.json для немецкого).
Добавьте rtl: true в начало файла, если новый язык RTL.
Загрузка на платформу переводов (Transifex):
Загрузите новый файл <code>.json в проект BrainMessenger на Transifex. Система автоматически определит, какие строки нуждаются в переводе.
Организация процесса перевода:
Назначьте переводчиков (профессионалов или проверенных участников сообщества, Принцип 14) для нового языка в Transifex.
Отвечайте на их вопросы по контексту строк.
Контролируйте прогресс перевода в Transifex.
Экспорт и Интеграция перевода:
После завершения перевода экспортируйте готовый файл <code>.json из Transifex.
Поместите его обратно в директорию frontend/src/locales/.
Импортируйте новый ресурс в файле инициализации i18next (frontend/src/index.ts) и добавьте его в объект resources.
Добавление опции выбора языка в UI:
Добавьте новый язык в список доступных языков в разделе "Настройки" > "Язык".
Тестирование:
Критически важный шаг (Принцип 3: Качество). Переключите язык в настройках приложения.
Проверьте:
Корректность перевода всех строк в UI.
Отсутствие обрезки текста (некоторые языки длиннее английского).
Корректное отображение UI для RTL-языков (зеркальность, выравнивание).
Локализованное форматирование дат и чисел.
Переведенные сообщения об ошибках с Backend.
Используйте сценарии тестирования, описанные в разделе 10.

8. Рекомендации: Лучшие Практики Локализации
8.1. Для Разработчиков (Пишем Код, Готовый к Миру)
Не хардкодьте строки! Все строки, которые видит пользователь, должны быть вынесены в файлы локализации и использоваться через функцию t('ключ') (Принцип 9: Система).
Используйте интерполяцию для строк с переменными данными (t('chat.newMessageNotification', { name: user.name })), а не конкатенацию строк в коде.
Учитывайте плюрализацию (формы единственного/множественного числа) при работе со счетчиками ({{count}} unread messages). i18next и react-intl поддерживают это.
Используйте react-intl для всех операций с датами, числами, валютами, чтобы их формат был корректным для каждой локали (Принцип 3: Качество, Принцип 2: Ценность).
Тестируйте UI с длинными переводами (например, немецкий, который часто длиннее английского) и RTL-языками, чтобы убедиться, что макет не ломается и текст не обрезается (Принцип 3: Качество).
8.2. Для Дизайнеров (Рисуем Макеты, Гибкие к Языкам)
Учитывайте, что текст на других языках может быть на 30% (и более) длиннее английского. Оставляйте достаточно пространства в компонентах.
Проектируйте макеты с учетом зеркального отображения для RTL. Иконки направлений, расположение текста и элементов должны меняться (Принцип 2: Доступность).
8.3. Для Переводчиков (Создаем Понятный Контент)
Сохраняйте контекст. Одно и то же английское слово может переводиться по-разному в зависимости от того, где оно используется (например, "Send" для сообщения или для заявки). Платформы вроде Transifex помогают добавлять контекстные комментарии.
Придерживайтесь последовательной терминологии, используйте глоссарий, если он есть.
Сохраняйте тон приложения (дружелюбный, но профессиональный, особенно в системных сообщениях и ошибках).
Используйте возможности интерполяции и плюрализации в файлах перевода.
9. Форматирование (Даты, Числа, Валюты): Детали, Создающие Качество (Принцип 3)
Корректное отображение локализованных данных критически важно для качества (Принцип 3) и удобства (Принцип 2).

Даты и Время:

import { FormattedDate, FormattedTime } from 'react-intl';

// Отобразит дату в формате, соответствующем текущей локали
<FormattedDate value={new Date()} /> // Например, "3/14/2025" (en-US) или "14.3.2025" (de-DE)
<FormattedDate value={new Date()} weekday="long" year="numeric" month="long" day="numeric" /> // "Friday, March 14, 2025"

// Отобразит время
<FormattedTime value={new Date()} /> // Например, "3:00:00 PM" (en-US) или "15:00:00" (de-DE)

import { FormattedNumber, FormattedCurrency } from 'react-intl';

// Отобразит число с локальным разделителем тысяч и десятичным знаком
<FormattedNumber value={1234.56} /> // "1,234.56" (en-US) или "1.234,56" (de-DE)

// Отобразит число как процент
<FormattedNumber value={0.75} style="percent" /> // "75%"

// Отобразит число как валюту (требует кода валюты)
<FormattedCurrency value={123.45} currency="USD" /> // "$123.45" (en-US)

Для работы с react-intl необходимо обернуть ваше приложение в IntlProvider и загрузить соответствующие данные локали (react-intl/locale-data).

10. Тестирование Локализации: Гарантия Качества (Принцип 3, 9)
Тестирование — неотъемлемая часть системы обеспечения качества (Принцип 3, 9).
Сценарии Тестирования:
Переключение языка: Убедитесь, что смена языка в настройках мгновенно (или после перезапуска, если требуется I18nManager) применяет новый перевод и RTL-макет.
Отображение всех строк: Проверьте основные экраны и диалоги, чтобы убедиться, что все видимые пользователю строки переведены.
Длинный текст: Проверьте UI с языками, известными своей длиной (например, немецкий, русский), чтобы убедиться, что текст не обрезается, элементы не накладываются друг на друга.
RTL-интерфейс: Тщательно проверьте UI на арабском (или другом RTL-языке): направление текста, выравнивание элементов, отображение иконок направлений, положение скроллбаров.
Интерполяция и Форматирование: Проверьте экраны, где используются динамические данные (имена, счетчики, даты, числа), убедитесь, что они форматируются правильно для текущей локали.
Локализованные ошибки: Инициируйте различные ошибки (неправильный логин, отсутствие элемента), проверьте, что сообщения об ошибках приходят на выбранном языке.
Инструменты:
Ручное тестирование: Основной метод. Тестирование на реальных устройствах с разными локалями.
Автоматизированные тесты (Cypress/detox): Можно написать тесты, которые проверяют наличие определенных ключей перевода на странице или даже делают скриншоты UI для сравнения LTR/RTL макетов (Принцип 9: Система).
Критерии Успеха Тестирования:
100% пользовательских строк переведены.
Отсутствие обрезки или наложения текста в UI.
Корректное и полное RTL-отображение для соответствующих языков.
Правильное локализованное форматирование дат, чисел, валют.

**A:** Безопасность — наш приоритет. Да, все личные чаты в BrainMessenger используют сквозное шифрование для обеспечения **целостности** (Принцип 7) и конфиденциальности ваших сообщений.
