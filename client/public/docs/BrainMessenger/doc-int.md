# Документация Интеграций BrainMessenger (Построение Надежной Системы с Внешними Рычагами)

### 1. Введение: Интеграции как Часть Фундамента Системы

**Название проекта:** BrainMessenger

**Описание:** Этот документ описывает, как BrainMessenger интегрируется с ключевыми внешними сервисами. Мы выбираем эти интеграции **прагматично** (Принцип 12), рассматривая их как **рычаги** (принцип Маргулана), которые позволяют нам не изобретать колесо, а использовать **надежные и масштабируемые** (Принцип 8, 9) сторонние решения для ускорения разработки и обеспечения **качества** (Принцип 3) нашего продукта. Эти интеграции являются неотъемлемой частью нашего **технологического фундамента** (Принцип 8).

**Цель:** Обеспечить **надежное хранение данных и файлов, эффективную коммуникацию (уведомления, email) и безопасные платежи**, используя лучшие в своем классе внешние сервисы, чтобы BrainMessenger мог масштабироваться и быть **ценным АКТИВОМ** (Принцип 2, 10) для пользователей.

**Принципы, которыми руководствуется выбор и настройка интеграций:**

*   **Система и Оптимизация (Принцип 9):** Интеграции должны быть частью единой, хорошо продуманной системы, где каждый компонент выполняет свою роль эффективно.
*   **Прагматизм и Реализм (Принцип 12):** Выбор сервисов с учетом текущих потребностей, бюджета (бесплатные/доступные тарифы на старте) и простоты интеграции.
*   **Долгосрочное Мышление (Принцип 8):** Выбор сервисов, способных масштабироваться вместе с ростом проекта.
*   **Качество > Количество (Принцип 3):** Фокус на надежности и производительности интеграций, даже если их список на старте небольшой.
*   **Создание Ценности (Принцип 2):** Каждая интеграция должна добавлять реальную ценность для конечного пользователя (например, надежное хранение файлов, мгновенные уведомления).

### 3. Интеграция с Neon: Фундамент Структурированных Данных

#### 3.1. Общая Информация

*   **Описание:** Neon предоставляет управляемый сервис PostgreSQL. Мы используем его как **основное хранилище структурированных данных**, таких как информация о пользователях, чатах, сообщениях и метаданных файлов.
*   **Преимущества (Выбор по Принципам):**
    *   **Прагматизм (P12):** Бесплатный тариф на старте (30 ГБ) полностью покрывает начальные потребности.
    *   **Система и Долгосрочное Мышление (P9, P8):** Управляемый сервис снижает операционную нагрузку, автоматическое масштабирование и репликация доступны на платных планах, обеспечивая готовность к росту.
    *   **Качество (P3):** Надежность и производительность PostgreSQL как проверенной СУБД.
*   **Роль:** Ядро, к которому обращается Backend для всех CRUD-операций над метаданными.

#### 3.2. Конфигурация: Подключение к Системе Данных

1.  **Регистрация и Создание Проекта:**
    *   Создайте аккаунт на `https://neon.tech`.
    *   Создайте новый проект (например, `brainmessenger-prod`) в консоли Neon.
2.  **Настройка Базы Данных:**
    *   Neon автоматически предоставляет инстанс PostgreSQL.
    *   **Примените схему базы данных:** Используйте Prisma Migrate для инициализации структуры таблиц (см. `database/schema.prisma` и Руководство по разработке).
3.  **Переменные Окружения:**
    *   Получите **строку подключения** (Connection Details) из консоли Neon (Settings → Connection Details).
    *   Добавьте в ваш `.env` файл:
        ```env
        # Строка подключения к базе данных Neon
        DATABASE_URL=postgresql://<user>:<password>@<neon-host>/<dbname>?sslmode=require
        ```
    *   *Примечание:* Используйте `sslmode=require` для безопасного соединения.
4.  **Подключение Backend (NestJS + Prisma):**
    *   Prisma использует `DATABASE_URL` для подключения. Убедитесь, что Prisma Client сгенерирован (`npx prisma generate`).
    *   Используйте `PrismaService` (из `nestjs-prisma`) для всех операций с БД.
        ```typescript
        // backend/src/app.module.ts (или db.module.ts)
        import { Module } from '@nestjs/common';
        import { PrismaService } from 'nestjs-prisma'; // Убедитесь, что PrismaModule импортирован в app.module

        @Module({
          providers: [PrismaService], // PrismaService доступен для инъекции
          exports: [PrismaService],
        })
        export class DbModule {}
        ```

#### 3.3. Методы и Сценарии Взаимодействия: API Данных

Все взаимодействия с Neon происходят через Prisma ORM в Backend, что обеспечивает **типизацию, безопасность и системность** (Принцип 3, 9).

*   **Сценарий: Регистрация пользователя:**
    *   **Взаимодействие:** Backend получает данные пользователя → Использует `PrismaService` для создания новой записи в таблице `users`.
    *   **Пример кода (Backend Service):**
        ```typescript
        // backend/src/auth/auth.service.ts
        import { Injectable } from '@nestjs/common';
        import { PrismaService } from 'nestjs-prisma';
        import { Prisma } from '@prisma/client'; // Импорт типов Prisma

        @Injectable()
        export class AuthService {
          constructor(private prisma: PrismaService) {}

          async signUp(data: Prisma.UserCreateInput) {
            // Пример: пароль должен быть захэширован ДО сохранения
            const hashedPassword = await hashPassword(data.password); // Использование утилиты для хэширования
            return this.prisma.user.create({
              data: {
                email: data.email,
                name: data.name,
                password: hashedPassword, // Сохраняем захэшированный пароль
                // ... другие поля по схеме
              },
            });
          }
        }
        ```
    *   **Результат:** Данные пользователя надежно сохранены в Neon.
*   **Сценарий: Получение списка чатов пользователя:**
    *   **Взаимодействие:** Frontend запрашивает чаты через GraphQL API → Backend использует `PrismaService` для выполнения запроса к таблице `chats` (возможно, с фильтрацией по `userId` и `include` для метаданных).
    *   **Пример кода (Backend Service):**
        ```typescript
        // backend/src/chat/chat.service.ts
        import { Injectable } from '@nestjs/common';
        import { PrismaService } from 'nestjs-prisma';

        @Injectable()
        export class ChatService {
          constructor(private prisma: PrismaService) {}

          async getUserChats(userId: string) {
            // Используем индексы @@index([userId]) для оптимизации (Принцип 9, Оптимизация)
            return this.prisma.chat.findMany({
              where: {
                users: { // Предполагается связь Many-to-Many с таблицей UserChat
                  some: {
                    userId: userId,
                  },
                },
              },
              select: { // Выбираем только необходимые поля (Принцип 9, Оптимизация)
                id: true,
                name: true,
                type: true,
                // ... другие нужные поля
              },
              orderBy: { // Сортировка по дате последнего сообщения для актуальности (Принцип 2, Ценность)
                lastMessageAt: 'desc',
              },
            });
          }
        }
        ```
*   **Сценарий: Сохранение сообщения:**
    *   **Взаимодействие:** Пользователь отправляет сообщение → Frontend отправляет его через GraphQL API → Backend использует `PrismaService` для создания новой записи в таблице `messages`, связывая ее с `chatId` и `userId`.
    *   **Пример кода (Backend Service):**
        ```typescript
        // backend/src/message/message.service.ts
        import { Injectable } from '@nestjs/common';
        import { PrismaService } from 'nestjs-prisma';
        import { Prisma } from '@prisma/client';

        @Injectable()
        export class MessageService {
          constructor(private prisma: PrismaService) {}

          async createMessage(data: Prisma.MessageCreateInput) {
            // Используем транзакцию, если нужно обновить дату последнего сообщения в чате
            return this.prisma.$transaction(async (tx) => {
              const message = await tx.message.create({ data });
              await tx.chat.update({
                where: { id: data.chatId },
                data: { lastMessageAt: message.createdAt }, // Обновляем время последнего сообщения
              });
              return message;
            });
          }
        }
        ```

#### 3.4. Обработка Ошибок Интеграции: Поддержание Качества Системы

Надежная система должна уметь обрабатывать ошибки внешних зависимостей (Принцип 3: Качество).

*   **Connection Error:** Невозможно установить соединение с Neon.
    *   **Причина:** Неверная строка подключения (`DATABASE_URL`), проблемы с сетью, недоступность сервиса Neon.
    *   **Решение:** Проверить переменную окружения `DATABASE_URL`, статус сервиса Neon. Логировать ошибку на Backend (Sentry/Winston).
*   **Database Query Error:** Ошибка выполнения запроса к БД (например, нарушение уникальности, неверный тип данных).
    *   **Причина:** Ошибка в логике Backend, несоответствие схемы БД.
    *   **Решение:** Отладить запрос в Backend, проверить логи Prisma. Логировать ошибку.
*   **Rate Limit Exceeded (на платных тарифах):** Превышены лимиты на количество запросов или вычислительные ресурсы в Neon.
    *   **Причина:** Высокая нагрузка, неоптимизированные запросы, недостаточный тарифный план.
    *   **Решение:** Оптимизировать запросы (см. Руководство по оптимизации), рассмотреть переход на более высокий тарифный план Neon.

### 4. Интеграция с Cloudflare R2: Хранилище Цифровых Активов

#### 4.1. Общая Информация

*   **Описание:** Cloudflare R2 — это сервис объектного хранилища, совместимый с S3 API. Мы используем его для хранения **всех файлов пользователей**, таких как изображения, видео, документы, а также зашифрованные данные.
*   **Преимущества (Выбор по Принципам):**
    *   **Прагматизм (P12), Создание Богатства/Активов (P10):** **Отсутствие платы за исходящий трафик** — критически важное преимущество для мессенджера, где пользователи активно обмениваются файлами. Бесплатный тариф 10 ГБ на старте.
    *   **Качество (P3):** Высокая производительность и надежность благодаря глобальной сети Cloudflare.
    *   **Долгосрочное Мышление (P8), Система (P9):** S3-совместимый API позволяет легко использовать стандартные библиотеки и инструменты, обеспечивая гибкость в будущем.
*   **Роль:** Децентрализованное хранилище для крупных бинарных объектов, снижающее нагрузку на основную базу данных.

#### 4.2. Конфигурация: Подключение к Хранилищу Активов

1.  **Создание Бакета:**
    *   Войдите в Cloudflare Dashboard → R2.
    *   Нажмите **"Create Bucket"**.
    *   Назовите бакет (например, `brainmessenger-files`). Название должно быть уникальным в рамках всех аккаунтов Cloudflare.
    *   **Настройки:** Включите шифрование в покое (Encryption at rest) — это часть нашей **системы безопасности** (Принцип 3, 5). Ограничьте публичный доступ к бакету, доступ будет осуществляться через подписанные URL (если требуется) или через Backend.
2.  **Ключи Доступа:**
    *   В Cloudflare Dashboard перейдите в R2 → Manage R2 API Tokens.
    *   Создайте новый API-токен. Предоставьте ему минимально необходимые разрешения (например, Read and Write to specific buckets).
    *   Скопируйте `Access Key ID` и `Secret Access Key`.
3.  **Переменные Окружения:**
    *   Добавьте в ваш `.env` файл:
        ```env
        # Cloudflare R2 Credentials
        R2_ENDPOINT=<Your R2 Endpoint> # Формат: https://<account-id>.r2.cloudflarestorage.com
        R2_ACCESS_KEY=<Your Access Key ID>
        R2_SECRET_KEY=<Your Secret Access Key>
        R2_BUCKET=brainmessenger-files # Название вашего бакета
        
        # Ключ шифрования для конфиденциальных данных (важно для P3, P5, P10)
        ENCRYPTION_KEY=<Your 32-byte (256-bit) hex or base64 encryption key> # Сгенерируйте надежный ключ!
        ```
    *   *Примечание:* **Не коммитьте эти ключи в Git!** (Принцип 3, 5, 6). Используйте `.env.example` без чувствительных данных. **ENCRYPTION_KEY** должен быть сгенерирован надежно и храниться в безопасности.
4.  **Подключение Backend (`@aws-sdk/client-s3`):**
    *   R2 совместим с S3 API, поэтому можно использовать стандартный AWS SDK для S3.
    *   Установите SDK: `npm install @aws-sdk/client-s3`.
    *   Инициализируйте S3 Client в вашем Backend-сервисе (например, `StorageService`).
        ```typescript
        // backend/src/storage/storage.service.ts
        import { Injectable } from '@nestjs/common';
        import { S3Client, PutObjectCommand, GetObjectCommand } from '@aws-sdk/client-s3';
        import { getSignedUrl } from '@aws-sdk/s3-request-presigner'; // Для подписанных URL
        import { createCipheriv, randomBytes, createDecipheriv } from 'crypto'; // Для шифрования

        @Injectable()
        export class StorageService {
          private r2: S3Client;
          private encryptionKey: Buffer; // Ключ для шифрования

          constructor() {
            this.r2 = new S3Client({
              region: 'auto', // R2 не использует регионы в классическом понимании
              endpoint: process.env.R2_ENDPOINT,
              credentials: {
                accessKeyId: process.env.R2_ACCESS_KEY,
                secretAccessKey: process.env.R2_SECRET_KEY,
              },
            });
            // Важно: ENCRYPTION_KEY должен быть Buffer 32 байт для aes-256-cbc
            this.encryptionKey = Buffer.from(process.env.ENCRYPTION_KEY, 'hex'); // Или 'base64'
            if (this.encryptionKey.length !== 32) {
                 // Логировать ошибку или выбросить исключение - ключ неверной длины
                 console.error("Invalid ENCRYPTION_KEY length. Must be 32 bytes for aes-256-cbc.");
                 // process.exit(1); // Возможно, стоит остановить приложение
            }
          }

          // Утилита для шифрования (часть системы безопасности - P3, P5)
          private encrypt(data: string): { encrypted: string, iv: string } {
            const iv = randomBytes(16); // Initialization Vector, 16 байт для aes-256-cbc
            const cipher = createCipheriv('aes-256-cbc', this.encryptionKey, iv);
            let encrypted = cipher.update(data, 'utf8', 'hex');
            encrypted += cipher.final('hex');
            return { encrypted, iv: iv.toString('hex') };
          }

          // Утилита для дешифрования
          private decrypt(encrypted: string, iv: string): string {
              const decipher = createDecipheriv('aes-256-cbc', this.encryptionKey, Buffer.from(iv, 'hex'));
              let decrypted = decipher.update(encrypted, 'hex', 'utf8');
              decrypted += decipher.final('utf8');
              return decrypted;
          }

          // ... методы для работы с R2 (загрузка, скачивание, удаление)
        }
        ```

#### 4.3. Методы и Сценарии Взаимодействия: API Хранилища

Взаимодействие с R2 осуществляется через Backend, что позволяет контролировать доступ, обрабатывать файлы (например, оптимизация изображений перед загрузкой) и управлять метаданными в БД (Принцип 9: Система).

*   **Сценарий: Загрузка файла пользователем:**
    *   **Взаимодействие:** Пользователь прикрепляет файл в чате → Frontend отправляет файл на Backend → Backend (`StorageService`) загружает файл в R2 → Backend сохраняет метаданные файла (URL, тип, размер) в Neon → Backend отправляет сообщение в чат с ссылкой на файл.
    *   **Пример кода (Backend Service):**
        ```typescript
        // backend/src/storage/storage.service.ts (продолжение)
        // ... S3Client и encryptionKey инициализация ...

        async uploadFile(fileBuffer: Buffer, fileName: string, mimeType: string): Promise<string> {
          const key = `uploads/${Date.now()}-${fileName}`; // Уникальный ключ файла
          const params = {
            Bucket: process.env.R2_BUCKET,
            Key: key,
            Body: fileBuffer,
            ContentType: mimeType, // Важно для отдачи файла
          };
          const command = new PutObjectCommand(params);
          await this.r2.send(command); // Асинхронная загрузка

          // Возвращаем URL для сохранения в БД. R2 позволяет прямые ссылки, если настроен публичный домен.
          // ИЛИ генерируем подписанный URL для ограниченного доступа (более безопасно).
          // Пример прямой ссылки (требует настройки Public Access URL в CF Dashboard):
          // const publicUrl = `${process.env.R2_ENDPOINT.replace('.r2.cloudflarestorage.com', '.r2.dev')}/${process.env.R2_BUCKET}/${key}`;
          // Для своего домена (предпочтительно):
          const fileUrl = `https://r2.brainmessenger.com/${key}`; // Ваш настроенный домен Cloudflare Workers/Pages

          return fileUrl;
        }

        //backend/src/chat/chat.service.ts (интеграция uploadFile)
        // ... constructor с StorageService ...
        async sendFileMessage(chatId: string, userId: string, file: Express.Multer.File): Promise<Message> {
           // Опционально: обработка изображения перед загрузкой (см. DocOptimizationIn)
           // const processedFileBuffer = await this.imageProcessingService.processImage(file.buffer, 800, 600); // Пример
           const fileUrl = await this.storageService.uploadFile(file.buffer, file.originalname, file.mimetype);

           // Создаем сообщение в БД со ссылкой на файл
           return this.prisma.message.create({
             data: {
               chatId,
               userId,
               content: '[File]', // Или другое обозначение файла
               fileUrl: fileUrl, // Сохраняем ссылку на файл в БД
               // fileMetadata: { type: file.mimetype, size: file.size }, // Опционально: сохранение метаданных в JSON поле или отдельной таблице
             },
           });
        }
        ```
    *   **Результат:** Файл надежно хранится в R2, ссылка на него доступна в БД для отображения в чате.
*   **Сценарий: Хранение конфиденциальных данных (например, зашифрованные метаданные):**
    *   **Взаимодействие:** При регистрации или обновлении профиля Backend шифрует чувствительные данные (например, резервные email, секретные вопросы) и сохраняет их в R2 в отдельном бакете или папке.
    *   **Пример кода (Backend Service):**
        ```typescript
        // backend/src/storage/storage.service.ts (продолжение)
        // ... методы encrypt/decrypt ...

        async storeSensitiveData(userId: string, sensitiveInfo: any): Promise<string> {
           const encryptedData = {};
           for (const key in sensitiveInfo) {
               if (sensitiveInfo.hasOwnProperty(key)) {
                   const { encrypted, iv } = this.encrypt(JSON.stringify(sensitiveInfo[key])); // Шифруем каждое поле или весь объект
                   encryptedData[key] = { encrypted, iv };
               }
           }

           const key = `sensitive/${userId}/metadata.json`;
           const params = {
               Bucket: process.env.R2_BUCKET, // Или отдельный бакет для чувствительных данных
               Key: key,
               Body: JSON.stringify(encryptedData),
               ContentType: 'application/json',
           };
           const command = new PutObjectCommand(params);
           await this.r2.send(command);
           return `https://r2.brainmessenger.com/${key}`; // Ссылка для внутреннего использования
        }

        // backend/src/user/user.service.ts (интеграция storeSensitiveData)
        // ... constructor с StorageService ...
        async createUserWithSensitiveData(userData: any, sensitiveData: any) {
            // Создаем пользователя в БД
            const user = await this.prisma.user.create({ data: userData });
            // Сохраняем зашифрованные данные в R2
            const sensitiveDataUrl = await this.storageService.storeSensitiveData(user.id, sensitiveData);
            // Сохраняем ссылку на зашифрованные данные в БД (опционально)
            await this.prisma.user.update({
                where: { id: user.id },
                data: { sensitiveDataUrl: sensitiveDataUrl } // Добавить поле sensitiveDataUrl в схему Prisma
            });
            return user;
        }
        ```
    *   **Результат:** Конфиденциальные данные надежно зашифрованы и хранятся отдельно от основной БД в R2. Доступ к ним возможен только через Backend с использованием ключа шифрования.
*   **Сценарий: Получение файла (для отображения или скачивания):**
    *   **Взаимодействие:** Frontend получает URL файла из метаданных сообщения (из Neon) → Frontend (или Backend, если требуется аутентификация/подписанные URL) использует этот URL для скачивания файла напрямую из R2.
    *   **Реализация:** Если бакет R2 настроен с Public Access URL (через Cloudflare Workers/Pages), Frontend может обращаться напрямую. Если доступ ограничен, Backend может генерировать временные подписанные URL через `@aws-sdk/s3-request-presigner`.
        ```typescript
        // backend/src/storage/storage.service.ts (продолжение)
        // ... S3Client инициализация ...

        // Метод для генерации подписанного URL (более безопасный подход для приватных файлов)
        async getSignedFileUrl(fileKey: string): Promise<string> {
            const command = new GetObjectCommand({
                Bucket: process.env.R2_BUCKET,
                Key: fileKey,
            });
            // Сгенерировать подписанный URL, действительный, например, 60 секунд
            const signedUrl = await getSignedUrl(this.r2, command, { expiresIn: 60 });
            return signedUrl;
        }
        ```

#### 4.4. Обработка Ошибок Интеграции: Хранилище Активов

*   **403 Forbidden:** Ошибка доступа к бакету R2.
    *   **Причина:** Неверные ключи доступа (`R2_ACCESS_KEY`, `R2_SECRET_KEY`), неверные разрешения токена, попытка доступа к приватному файлу без аутентификации/подписанного URL.
    *   **Решение:** Проверить переменные окружения, разрешения токена R2. Убедиться, что логика доступа (публичный URL vs подписанный URL) реализована корректно. Логировать ошибку.
*   **404 Not Found:** Файл отсутствует в R2.
    *   **Причина:** Неверный ключ файла (`Key`), файл был удален.
    *   **Решение:** Проверить ключ файла, который был сохранен в Neon. Убедиться, что файл действительно был загружен. Логировать ошибку.
*   **429 Too Many Requests:** Превышен лимит запросов к R2 API.
    *   **Причина:** Высокая частота запросов загрузки/скачивания.
    *   **Решение:** Внедрить Rate Limiting на Backend (см. Руководство по оптимизации). Оптимизировать логику работы с файлами.
*   **Encryption/Decryption Error:** Ошибка шифрования/дешифрования.
    *   **Причина:** Неверный ключ шифрования (`ENCRYPTION_KEY`), поврежденный IV, поврежденные зашифрованные данные.
    *   **Решение:** Убедиться, что используется правильный ключ (`ENCRYPTION_KEY`), который совпадает с ключом, использованным при шифровании. Проверить целостность данных и IV. Логировать критическую ошибку.

### 5. Взаимодействие Neon и Cloudflare R2: Оптимизация Хранения Данных (Принцип 9, 12)

**Мы используем Neon и R2 совместно, чтобы создать оптимизированную систему хранения данных**, где каждый сервис выполняет свою лучшую функцию (Принцип 9).

*   **Neon (PostgreSQL):** Хранит **структурированные метаданные**, которые часто запрашиваются и по которым выполняются сложные запросы (пользователи, чаты, текстовые сообщения, ссылки на файлы).
*   **Cloudflare R2:** Хранит **неструктурированные бинарные данные** большого размера (файлы), которые реже запрашиваются и не требуют сложной логики выборки по содержимому.

**Сценарий комплексного взаимодействия:**

1.  Пользователь отправляет файл через чат (Frontend -> Backend).
2.  Backend (`StorageService`) **загружает файл в Cloudflare R2** (используя S3 API). Получает URL файла (или ключ).
3.  Backend (`ChatService`) **сохраняет метаданные сообщения в Neon через Prisma**, включая полученный URL/ключ файла.
4.  Когда другой пользователь открывает чат, Backend (`ChatService`) **получает список сообщений из Neon через Prisma**, включая URL/ключи файлов.
5.  Frontend отображает список сообщений и использует URL/ключи для **скачивания файлов напрямую из R2** (или через Backend, если требуется аутентификация).

**Преимущества такого разделения:**

*   **Оптимизация производительности БД:** PostgreSQL не "засоряется" большими бинарными данными, что улучшает скорость текстовых запросов (Принцип 9).
*   **Снижение стоимости:** R2 предлагает очень выгодные тарифы для хранения больших объемов данных без платы за исходящий трафик, что **прагматично** (Принцип 12) и способствует созданию **актива** (Принцип 10).
*   **Масштабируемость:** Каждый сервис масштабируется независимо в соответствии со своим типом нагрузки (Neon для транзакций, R2 для объема/трафика файлов).
*   **Надежность:** Разделение ответственности между сервисами повышает общую отказоустойчивость системы.

### 6. Другие Интеграции: Расширение Функционала Системы (Принцип 2, 12)

Эти интеграции добавляют **ценность** (Принцип 2) и используют **прагматичные, готовые решения** (Принцип 12).

#### 6.1. Firebase (Push-уведомления)

*   **Назначение:** Мгновенная доставка уведомлений о новых сообщениях, звонках и системных событиях пользователям мобильных приложений.
*   **Роль:** **Критически важный компонент системы уведомлений** (Принцип 9) для поддержания вовлеченности пользователей и оперативного информирования (Принцип 2).
*   **Конфигурация:**
    *   Переменная окружения: `FIREBASE_CREDENTIALS_PATH` (путь к JSON-файлу учетных данных сервисного аккаунта Firebase). **Храните этот файл безопасно!**
    *   Используется библиотека `firebase-admin` в Backend NestJS.
*   **Пример использования:** Отправка уведомления при получении нового сообщения (см. Руководство по оптимизации для батчинга и тем).

#### 6.2. Stripe (Платежи)

*   **Назначение:** Обработка подписок на Premium-функционал BrainMessenger.
*   **Роль:** **Финансовый рычаг** для поддержки развития проекта (Принцип 10: Создание Богатства) и предоставления дополнительной **ценности** (Принцип 2) премиум-пользователям.
*   **Конфигурация:**
    *   Переменная окружения: `STRIPE_SECRET_KEY`.
    *   Используется библиотека `stripe` в Backend NestJS.
*   **Пример использования:** Создание сессии оплаты при попытке оформить подписку. Обработка вебхуков для подтверждения успешной оплаты.

#### 6.3. Gmail API (Отправка Email)

*   **Назначение:** Отправка транзакционных email, таких как коды подтверждения для регистрации и двухфакторной аутентификации (2FA), уведомления о сбросе пароля.
*   **Роль:** **Критически важный компонент системы аутентификации и безопасности** (Принцип 9, 5).
*   **Конфигурация:** Требует настройки сервисного аккаунта Google Cloud и получения учетных данных (см. Техническую документацию, раздел 10.1).
    *   Переменные окружения для учетных данных сервисного аккаунта или API-ключей.
*   **Пример использования:** Вызов API для отправки письма с кодом подтверждения после ввода email при регистрации или включении 2FA (см. Спецификацию API, раздел 2.4).

### 7. Рекомендации по Интеграциям: Поддержание Здоровья Системы (Принцип 3, 5, 9)

Чтобы интеграции оставались **надежными, безопасными и эффективными** (Принцип 3, 9), следуйте этим рекомендациям, связанным с поддержанием "здоровья" системы (Принцип 5, применительно к технике):

*   **Безопасность:**
    *   **Никогда не коммитьте ключи и секреты в Git.** Используйте переменные окружения и системы управления секретами в продакшен-окружении (например, Kubernetes Secrets, HashiCorp Vault).
    *   Регулярно проводите **ротацию API-токенов** (R2, Firebase, Gmail, Stripe) для повышения безопасности.
    *   Используйте **принцип минимальных привилегий** при создании API-токенов: давайте им только те разрешения, которые действительно необходимы.
*   **Масштабирование:**
    *   **Мониторьте использование ресурсов** (место в Neon/R2, запросы к API Firebase/Stripe) через консоли соответствующих сервисов. Будьте готовы перейти на платный тариф при росте нагрузки (Принцип 8, 12).
    *   Используйте **оптимизационные подходы** (например, батчинг запросов к Firebase, индексирование в Neon, см. Руководство по оптимизации), чтобы отложить необходимость перехода на более дорогие тарифы.
*   **Мониторинг и Логирование:**
    *   Настройте **логирование ошибок интеграций** в NestJS (Winston, Sentry).
    *   Используйте **системы мониторинга** (Prometheus/Grafana) для отслеживания метрик запросов к внешним сервисам (задержки, количество ошибок) — это часть **Кайдзен** (Принцип 9), позволяющая выявлять проблемы на ранней стадии.
*   **Обработка Ошибок:**
    *   Реализуйте **грамотную обработку ошибок** от внешних сервисов на Backend. Логируйте их подробно.
    *   Для критически важных операций (например, регистрация, платежи) предусмотрите механизмы **повторных попыток (retries)** или **компенсирующих действий** (например, отметка платежа как незавершенного) при временных сбоях внешних сервисов.
