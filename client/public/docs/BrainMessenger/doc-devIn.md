# Руководство по Разработке BrainMessenger (Применение Принципов в Коде)

### 1. Введение: Процесс Разработки как Система Совершенствования

**Название проекта:** BrainMessenger

**Описание:** Этот документ является вашим путеводителем по процессу разработки BrainMessenger. Мы строим этот проект не хаотично, а как **целостную систему** (Принцип 9), где каждый участник команды (или вы сами) действует в соответствии с **четкими стандартами и принципами**, стремясь к **постоянному совершенствованию** (Принцип 1, 9). Этот процесс напрямую влияет на **качество** (Принцип 3) конечного продукта и его превращение в **ценный АКТИВ** (Принцип 10).

**Цель:** Обеспечить **единый, эффективный и масштабируемый** процесс разработки, упростить онбординг новых участников и поддерживать высокий уровень **качества кода и архитектуры** на каждом этапе, включая глубокое изучение и применение технологий (как видно по трекеру видео).

**Аудитория:** Все, кто пишет код или управляет процессом разработки BrainMessenger (Frontend, Backend, Mobile, Desktop, DevOps разработчики).

**Принципы, которыми руководствуется процесс разработки:**

*   **Система и Оптимизация (Принцип 9):** Структура репозитория, стандарты кодирования, CI/CD, использование инструментов — все направлено на создание эффективной и предсказуемой системы. **Кайдзен** применяется к самому процессу разработки.
*   **Качество > Количество (Принцип 3):** Приоритет отдается написанию чистого, тестируемого и поддерживаемого кода, даже если это занимает больше времени на старте.
*   **Постоянное Обучение (Принцип 1):** Процесс разработки включает изучение новых технологий, их прагматичное применение и рефлексию над полученным опытом (как документируется в видео трекере и Кайдзен-часах).
*   **Прагматизм и Реализм (Принцип 12):** Выбор инструментов и подходов, подходящих для текущего этапа проекта, без излишней сложности.
*   **Долгосрочное Мышление (Принцип 8):** Архитектурные решения принимаются с прицелом на будущее масштабирование и поддерживаемость. Код пишется так, чтобы его было легко развивать.
*   **Ответственность и Проактивность (Принцип 6):** Каждый разработчик несет ответственность за качество своего кода, тесты и активно участвует в улучшении процессов.

### 2. Структура Репозитория: Монорепо как Система Организации Кода

Проект организован как **монорепозиторий**, используя **Turborepo** для управления пакетами. Это — сознательный выбор, призванный повысить **эффективность** (Принцип 9) за счет переиспользования кода и **оптимизации** (Принцип 9) процессов сборки и тестирования.

```
BrainMessenger/
├── packages/               # Все отдельные пакеты/приложения
│   ├── core/               # Общий переиспользуемый код (логика, утилиты, API-клиент)
│   │   ├── src/            # Исходный код core
│   │   │   ├── api/        # Клиент для GraphQL API (Apollo Client)
│   │   │   ├── hooks/      # Общие React/React Native хуки
│   │   │   └── utils/      # Общие утилиты (форматирование, валидация и т.д.)
│   │   └── package.json    # Зависимости core
│   ├── mobile-desktop/     # Приложение для Android и Windows (React Native)
│   │   ├── src/            # Исходный код приложения
│   │   │   ├── components/ # Переиспользуемые UI-компоненты
│   │   │   ├── screens/    # Экраны приложения
│   │   │   ├── theme/      # Файлы тем (Light/Dark Mode, см. DocDesign)
│   │   │   └── platform/   # Платформо-специфичный код (Android, Windows)
│   │   │       ├── android/
│   │   │       └── windows/
│   │   ├── android/        # Нативные проекты Android
│   │   ├── windows/        # Нативные проекты Windows (C++)
│   │   ├── assets/         # Статические ресурсы (изображения, шрифты, звуки)
│   │   └── package.json
│   └── web/                # Веб-приложение (Next.js)
│       ├── pages/          # Страницы Next.js
│       ├── components/     # Компоненты React
│       ├── styles/         # Стили (Tailwind CSS)
│       └── package.json
├── backend/                # Серверная часть (NestJS)
│   ├── src/                # Исходный код backend
│   ├── test/               # Тесты (модульные, интеграционные)
│   └── package.json        # Зависимости backend
├── infrastructure/         # Конфигурации инфраструктуры (Terraform, Kubernetes)
│   ├── k8s/                # Манифесты Kubernetes
│   └── main.tf             # Terraform-конфигурация
├── docs/                   # Проектная документация (эти файлы)
├── .github/workflow/       # Конфигурации GitHub Actions (CI/CD)

└── .env.example            # Пример переменных окружения
```

*   **`core`:** Содержит код, который является **ядром системы** (Принцип 9) и может быть использован на любой платформе (API-клиент, общие утилиты, бизнес-логика, не связанная с UI/сервером). Переиспользование повышает **эффективность** (Принцип 9) и **согласованность** (Принцип 7).
*   **`mobile-desktop`:** Содержит код для **клиентского Актива** (Принцип 10) на мобильных и десктопных платформах.
*   **`web`:** Содержит код для **клиентского Актива** на веб-платформе.
*   **`backend`:** Содержит код **серверной части системы**, обрабатывающей запросы, взаимодействующей с БД и внешними сервисами. Это критически важный **рычаг** (Маргулан) для работы всего приложения.
*   **`infrastructure`:** Содержит конфигурации для **развертывания системы** (Принцип 9) на облачных ресурсах.
*   **`docs`:** Содержит **документацию — часть системы знаний** (Принцип 1), необходимую для эффективной работы и онбординга.
*   **`turbo.json`:** Определяет, как Turborepo управляет задачами (сборка, тестирование, линтинг) по всему монорепо, используя **кэширование и параллельное выполнение** для **оптимизации** (Принцип 9) времени разработки.

### 4. Стандарты Кодирования: Дисциплина как Основа Качества (Принцип 3, 9)

Соблюдение стандартов кодирования — это проявление **дисциплины** и часть **системы обеспечения качества**. Это делает код предсказуемым, легко читаемым и поддерживаемым другими разработчиками (или вами самими в будущем).

#### 4.1. Общие Принципы

*   **DRY (Don’t Repeat Yourself):** Избегайте дублирования кода. Используйте общие функции, компоненты, хуки в пакете `core`.
*   **KISS (Keep It Simple, Stupid):** Стремитесь к простоте решений. Не усложняйте без необходимости. **Упрощать сложное** (Маргулан) применимо и к коду.
*   **Именование:** Используйте английский язык для всех идентификаторов (переменные, функции, классы), комментариев. Будьте последовательны.
*   **Комментарии:** Обязательны для:
    *   Сложной бизнес-логики (например, обработка платежей, управление WebSocket-соединениями).
    *   Интеграций с внешними сервисами (где могут быть неочевидные детали API).
    *   Критических участков кода (аутентификация, шифрование, работа с множествами).
    *   Неочевидных решений или обходных путей (с объяснением почему).

#### 4.2. Конвенции Именования

*   **Переменные и функции:** `camelCase` (например, `getUserProfile`, `chatMessage`).
*   **Классы и модули:** `PascalCase` (например, `UserService`, `ChatModule`).
*   **Константы:** `UPPER_SNAKE_CASE` (например, `MAX_FILE_SIZE`, `DEFAULT_TIMEOUT`).
*   **Файлы и папки:** `kebab-case` (например, `user-service.ts`, `chat-screen.tsx`, `image-processing`).
*   **API Endpoints (в контексте GraphQL):** Резолверы и их поля именуются согласно доменной модели, часто следуя REST-подобной логике (например, query `getUser`, mutation `sendMessage`).

#### 4.3. Форматирование и Линтинг: Автоматизация Поддержания Порядка

Используйте автоматические инструменты для обеспечения единообразного форматирования и соблюдения базовых правил кодирования. Это часть **системы автоматизации качества** (Принцип 9).

*   **Форматирование:** **Prettier** — автоматический форматтер кода.
    *   Настройте `.prettierrc` в корне проекта. Рекомендуемые настройки: 2 пробела для отступов, отсутствие точек с запятой в конце строк, одинарные кавычки для строк.
        ```json
        {
          "tabWidth": 2,
          "semi": false,
          "singleQuote": true,
          "trailingComma": "es5"
        }
        ```
*   **Линтинг:** **ESLint** — статический анализатор кода для выявления проблем.
    *   Используйте конфигурацию вроде Airbnb или Standard, адаптированную для TypeScript, React/React Native, NestJS. Настройте правила в `.eslintrc.js`.
    *   Интегрируйте ESLint в ваш редактор кода для получения обратной связи в реальном времени.

#### 4.4. Структура Кода: Модульность и Четкие Границы

Код организован модульно внутри каждого пакета, следуя принципам **системности** (Принцип 9) и разделения ответственности.

*   **Core (`packages/core`):**
    *   `api/`: Код для взаимодействия с GraphQL API BrainMessenger (Apollo Client).
    *   `hooks/`: Переиспользуемые хуки для React/React Native.
    *   `utils/`: Чистые функции-утилиты, не зависящие от конкретной платформы.
*   **Mobile-Desktop (`packages/mobile-desktop`):**
    *   `components/`: Небольшие, переиспользуемые UI-компоненты (`Button`, `MessageBubble`).
    *   `screens/`: Компоненты, представляющие целые экраны (`ChatScreen`, `SettingsScreen`).
    *   `theme/`: Файлы с цветовыми схемами и типографикой, определенные в DocDesign.
    *   `platform/`: Код, специфичный для Android или Windows (например, работа с нативными модулями).
*   **Web (`packages/web`):**
    *   `pages/`: Компоненты страниц Next.js.
    *   `components/`: UI-компоненты для веба (могут переиспользовать логику из `core/hooks` или `core/utils`).
    *   `styles/`: Файлы стилей (Tailwind CSS).
*   **Backend (`backend`):**
    *   Структура NestJS: Модули (`auth`, `user`, `chat`, `message`), сервисы (`AuthService`, `UserService`), контроллеры (для REST, если есть) или резолверы (для GraphQL), DTO (Data Transfer Objects), Entities (определенные Prisma).
    *   `src/`: Исходный код.
    *   `test/`: Тесты.

### 5. Используемые Библиотеки: Выбор Правильных Инструментов (Принцип 12)

Мы выбираем библиотеки, которые решают задачи **прагматично** (Принцип 12), обеспечивают **качество** (Принцип 3) и хорошо интегрируются в нашу **систему** (Принцип 9).

#### 5.1. Общие (Core)

*   `@apollo/client` (и связанные): Клиент GraphQL для выполнения запросов к Backend API. Используется в `core` и импортируется в `mobile-desktop` и `web`. Обеспечивает **эффективность** (Принцип 9) за счет управления кэшем.

#### 5.2. Mobile-Desktop (React Native)

*   `react-native-reanimated`, `moti`, `lottie-react-native`: Библиотеки для создания **плавных и производительных анимаций** (см. DocDesign). Связано с Принципом 3 (Качество UI) и Принципом 5 (Здоровье - снижение нагрузки).
*   `@react-navigation/native` (и связанные): Навигация между экранами приложения. **Системное** (Принцип 9) управление переходами.
*   `react-native-sound`: Базовая библиотека для воспроизведения звуков (сообщения, звонки). См. Документацию звуков.

#### 5.3. Web (Next.js)

*   `next`, `react`, `react-dom`: Основа веб-приложения. Next.js обеспечивает **систему** (Принцип 9) для SSR, SSG, маршрутизации и **оптимизации** (см. DocOptimizationIn).
*   `tailwindcss`: Утилитарный CSS-фреймворк для быстрой и **прагматичной** (Принцип 12) стилизации.
*   `framer-motion`: Библиотека для **качественных** (Принцип 3) анимаций в вебе.

#### 5.4. Backend (NestJS)

*   `@nestjs/*`: Основные пакеты NestJS (core, common, platform-express/fastify, graphql, typeorm/prisma, jwt, passport, websockets и т.д.). Обеспечивают **модульную, масштабируемую систему** (Принцип 9).
*   `@nestjs/prisma`: Интеграция NestJS и Prisma.
*   `graphql`, `@nestjs/graphql`, `apollo-server-express`: Реализация GraphQL API.
*   `@prisma/client`: Клиент Prisma ORM для взаимодействия с Neon (PostgreSQL).
*   `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`: Работа с Cloudflare R2 (S3-совместимый).
*   `ioredis`: Клиент Redis для кэширования и других задач.
*   `kafkajs`: Клиент Kafka для асинхронных задач.
*   `firebase-admin`: SDK для Firebase (Push-уведомления).
*   `stripe`: SDK для Stripe (Платежи).
*   `googleapis`: SDK для Google APIs (Gmail API).
*   `bcrypt` или `argon2`: Библиотека для **безопасного** (Принцип 3, 5) хэширования паролей.

#### 5.5. Общие (Cross-Package)

*   `typescript`: Язык разработки.
*   `winston` или `pino`: Библиотека для **структурированного логирования** (Принцип 9).
*   `@sentry/node`, `@sentry/react`, `@sentry/react-native`: Клиенты Sentry для **отслеживания ошибок** в продакшене (часть системы **качества** и **Кайдзен**).
*   `core-js`, `regenerator-runtime`: Полифиллы для поддержки современных возможностей JavaScript в старых средах (например, новые методы Set в ES2024).
*   `zod` или `class-validator`/`class-transformer`: Библиотеки для **валидации данных**, обеспечения **качества** (Принцип 3) на входе в систему.

#### 5.6. Работа с Множествами (Set) в ES2024+: Пример Применения Новых Знаний (Принцип 1)

**Постоянное обучение** (Принцип 1) и применение **прагматичных** (Принцип 12) инструментов языка — это часть нашей философии. Новые методы объекта `Set` в ECMAScript 2024 (поддержка в Node.js 22+, современных браузерах) предоставляют **более чистый, читаемый и часто более эффективный** (Принцип 9) способ работы с уникальными наборами данных по сравнению с ручными циклами или преобразованиями в массивы.

**Области применения в BrainMessenger:**
*   Управление списками пользователей в групповых чатах (добавление/удаление участников).
*   Отслеживание уникальных активных сессий или соединений WebSocket.
*   Управление правами доступа (проверка, входит ли набор разрешений в доступные).
*   Синхронизация списков (например, контактов, чатов) между клиентом и сервером, определение добавленных/удаленных элементов.

**Ключевые Методы (из ES2024):**
*   `union(iterable)`: Возвращает новое `Set` со всеми уникальными элементами из текущего `Set` и переданного итерируемого объекта.
*   `intersection(iterable)`: Возвращает новое `Set` с элементами, присутствующими в **обоих** множествах.
*   `difference(iterable)`: Возвращает новое `Set` с элементами, присутствующими в текущем `Set`, но **не** в переданном итерируемом объекте.
*   `symmetricDifference(iterable)`: Возвращает новое `Set` с элементами, которые есть в **одном**, но не в **обоих** множествах.
*   `isSubsetOf(iterable)`: Проверяет, являются ли все элементы текущего `Set` подмножеством переданного итерируемого объекта (`true` или `false`).
*   `isSupersetOf(iterable)`: Проверяет, содержит ли текущий `Set` все элементы переданного итерируемого объекта (`true` или `false`).
*   `isDisjointFrom(iterable)`: Проверяет, не имеют ли текущий `Set` и переданный итерируемый объект общих элементов (`true` или `false`).

**Примеры Использования в Коде (TypeScript):**

// backend/src/chat/chat.service.ts (пример управления участниками группы)
async updateGroupMembers(groupId: string, currentMemberIds: Set<string>, newMemberIds: Set<string>) {
  // Принцип 9: Оптимизация и ясность
  const addedMembers = newMemberIds.difference(currentMemberIds);
  const removedMembers = currentMemberIds.difference(newMemberIds);
  const unchangedMembers = currentMemberIds.intersection(newMemberIds); // Для проверки

  console.log('Added:', addedMembers);
  console.log('Removed:', removedMembers);

  // Выполнить операции добавления/удаления в БД через Prisma
  if (addedMembers.size > 0) {
    await this.prisma.userChat.createMany({
      data: Array.from(addedMembers).map(userId => ({ groupId, userId })),
    });
  }
  if (removedMembers.size > 0) {
    await this.prisma.userChat.deleteMany({
      where: {
        groupId: groupId,
        userId: { in: Array.from(removedMembers) },
      },
    });
  }

  // Проверка прав: Убедиться, что текущий пользователь имеет право изменять состав группы
  const requiredPermission = new Set(['manage_members']);
  const userPermissions = new Set(await this.getUserPermissions(userId, groupId)); // Получить права пользователя
  const hasPermission = requiredPermission.isSubsetOf(userPermissions); // Проверка через новый метод
  if (!hasPermission) {
      throw new ForbiddenException("Недостаточно прав для изменения состава группы.");
  }
}

**Рекомендации по использованию новых методов `Set`:**
*   **Применяйте их для операций с уникальными наборами данных.** Они разработаны специально для этого и часто более производительны, чем аналогичные операции с массивами, особенно для больших объемов данных.
*   **Учитывайте совместимость:** Новые методы доступны в Node.js 22+ и современных браузерах. Если вам нужно поддерживать старые среды, используйте полифиллы (`core-js/proposals/set-methods-v2`).
*   **Типизация в TypeScript:** Всегда указывайте тип элементов в `Set` (например, `Set<string>`, `Set<number>`) для лучшей проверки типов и автодополнения.

*   **Основная ветка разработки:** `develop`. Все новые функции и исправления интегрируются сюда.

    *   `feature/<название_фичи>`: Для разработки новой функциональности (например, `feature/graphql-auth`).
    *   `fix/<название_исправления>`: Для исправления ошибок (например, `fix/chat-message-scroll`).
    *   `refactor/<название>`: Для рефакторинга кода.
*   **Формат коммитов:** Используйте **Conventional Commits**. Это упрощает генерацию changelog и понимание цели каждого коммита.
    *   Пример:
        *   `feat(backend/auth): add JWT authentication` (новая функция в модуле аутентификации backend)
        *   `fix(mobile/chat): resolve message display bug on scroll` (исправление бага отображения сообщений в мобильном чате)
        *   `docs(roadmap): update Q3 2025 plan` (обновление документации)
        *   `chore(deps): update prisma to 5.x` (рутинная задача, обновление зависимости)
    *   Структура: `тип(scope): описание`
        *   `тип`: feat, fix, docs, chore, style, refactor, test, ci, perf, build
        *   `scope`: Необязательно, указывает пакет или область (`backend`, `web`, `mobile-desktop`, `core`, `auth`, `chat`, `ui`, `infra` и т.п.).

Тестирование — **неотъемлемая часть процесса обеспечения качества** (Принцип 3). Мы используем многоуровневый подход.

*   **Запуск всех тестов:**
    ```bash

    ```
*   **Виды тестов:**

    *   **Интеграционные тесты (Integration Tests):** Тестирование взаимодействия между несколькими компонентами (например, сервис -> БД, API -> сервис) (Jest). **Фокус:** Правильность связей между частями системы.

*   Настройте автоматический запуск тестов в CI/CD (GitHub Actions) на каждый PR и пуш в `develop`.

#### 7.4. Backend (NestJS)

*   **GraphQL API:** Следуйте принципам GraphQL (запрашивать только нужное). Используйте DataLoader для решения проблемы N+1 (см. DocOptimizationIn).
*   **Prisma:** Используйте ORM для всех взаимодействий с БД. Это обеспечивает **безопасность** (защита от SQL-инъекций - Принцип 3, 5) и **типизацию**.
*   **Сервисы:** Выносите основную бизнес-логику в сервисы. Контроллеры/Резолверы должны быть тонкими (только обработка запроса и вызов сервиса).
*   **Интеграции:** Взаимодействие с внешними сервисами (R2, Firebase, Stripe, Gmail API) должно быть инкапсулировано в отдельных сервисах (например, `StorageService`, `NotificationService`). Это делает код более **модульным и тестируемым** (Принцип 9).
*   **Обработка Ошибок:** Используйте фильтры исключений NestJS для централизованной обработки ошибок и возврата стандартизированных ответов API (см. DocSpec). Логируйте ошибки подробно (Winston, Sentry).

#### 7.5. Безопасность: Защита Вашего Актива (Принцип 3, 5, 10)

Безопасность — это не фича, а **фундаментальный аспект качества** (Принцип 3), напрямую связанный со **здоровьем системы** (Принцип 5) и защитой **цифрового актива** (Принцип 10: ваш продукт и данные пользователей).

*   **Аутентификация:** Используйте JWT для stateless аутентификации с короткоживущими access-токенами и refresh-токенами. **Надежно храните секрет JWT.**
*   **Хэширование Паролей:** **Никогда не храните пароли в открытом виде.** Используйте надежные алгоритмы хэширования (bcrypt, argon2).
*   **Валидация Ввода:** Строго валидируйте все входящие данные на Backend (например, с помощью `class-validator` и DTO).
*   **Защита от Атак:** Внедрите Rate Limiting (`@nestjs/throttler`) для защиты от перебора паролей или DDoS на уровне API. Используйте `helmet` для установки безопасных HTTP-заголовков.
*   **Шифрование Чувствительных Данных:** Шифруйте данные перед сохранением в R2 (см. DocInt). Надежно храните ключ шифрования отдельно от данных.
*   **2FA:** Реализуйте двухфакторную аутентификацию как опцию для пользователей.
