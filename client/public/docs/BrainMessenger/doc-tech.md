# Техническая Документация BrainMessenger (Система и Фундамент)

### 1. Общая Информация: Архитектура Цифрового Актива

**Название проекта:** BrainMessenger

**Описание:** BrainMessenger — это современный мессенджер, построенный как **надежная, масштабируемая и безопасная система** (Принцип 9), которая служит **ключевым цифровым АКТИВОМ** (Принцип 10). Этот документ предоставляет высокоуровневый обзор **технического фундамента** (Принцип 8) проекта, описывая основные компоненты, технологии и их взаимодействие.

**Цель:** Дать общее понимание структуры BrainMessenger, используемого стека технологий и принципов, лежащих в основе его построения, для всех членов команды и заинтересованных лиц.

**Принципы, отраженные в технической структуре:**

*   **Система и Оптимизация (Принцип 9):** Проект спроектирован как набор взаимосвязанных, оптимизированных компонентов.
*   **Долгосрочное Мышление (Принцип 8):** Выбор технологий и архитектуры с учетом будущих потребностей масштабирования и развития.
*   **Качество > Количество (Принцип 3):** Использование проверенных, надежных решений и фокус на качестве кода и инфраструктуры.
*   **Прагматизм и Реализм (Принцип 12):** Использование готовых сервисов (Neon, R2, Firebase) там, где это оправдано, вместо создания всего с нуля.
*   **Создание Ценности (Принцип 2):** Техническая структура направлена на обеспечение ключевых функций мессенджера (общение, файлы, звонки) на высоком уровне.

### 2. Технологический Стек: Инструментарий для Построения Системы

Мы используем современный и гибкий набор технологий, позволяющий эффективно строить и масштабировать BrainMessenger.

*   **Frontend (Клиентские Приложения):**
    *   **React Native (Android, Windows Desktop):** Единая кодовая база для мобильного и десктопного приложения. Выбран за кроссплатформенность и большой набор готовых библиотек.
    *   **Next.js (Web):** React-фреймворк для веб-приложения с поддержкой SSR/SSG для производительности и SEO.
    *   **TypeScript:** Строгая типизация для всего Frontend-кода. Повышает надежность и упрощает рефакторинг.
*   **Backend (Серверная Логика):**
    *   **Node.js:** Высокопроизводительная среда выполнения JavaScript/TypeScript.
    *   **NestJS:** Модульный фреймворк для Node.js. Обеспечивает структурированную архитектуру (модули, сервисы, контроллеры/резолверы) и поддержку TypeScript.
    *   **GraphQL:** Протокол API. Позволяет клиентам запрашивать точно те данные, которые нужны, снижая избыточность.
*   **База Данных и Кэширование:**
    *   **PostgreSQL (через Neon):** Надежная реляционная СУБД. Используется как основное хранилище структурированных данных (пользователи, чаты, сообщения, метаданные). Neon предоставляет управляемый сервис с функциями масштабирования и репликации.
    *   **Prisma ORM:** Инструмент для взаимодействия с PostgreSQL из NestJS. Обеспечивает типизацию и безопасность запросов.
    *   **Redis:** In-memory хранилище данных. Используется для кэширования, Rate Limiting, управления состоянием WebSocket-соединений (присутствие пользователей).
*   **Хранилище Файлов:**
    *   **Cloudflare R2:** Объектное хранилище, совместимое с S3 API. Используется для хранения файлов пользователей (фото, видео, документы, аватары, записи звонков), зашифрованных данных. Выбран за низкую стоимость и отсутствие платы за исходящий трафик.
*   **Асинхронная Обработка:**
    *   **Kafka:** Распределенная система очередей сообщений. Используется для надежной асинхронной обработки задач (например, обработка изображений, отправка уведомлений) вне основного потока API.
*   **Инфраструктура и Развертывание:**
    *   **Docker:** Контейнеризация приложений. Обеспечивает изоляцию и переносимость.
    *   **Kubernetes:** Оркестрация контейнеров. Управляет развертыванием, масштабированием и самовосстановлением сервисов в кластере.
    *   **Terraform:** Инфраструктура как код (IaC). Автоматизирует создание и управление облачными ресурсами.
*   **Тестирование, Мониторинг, Логирование:**
    *   **Jest, Cypress, Detox:** Инструменты для модульного, интеграционного и E2E-тестирования.
    *   **Prometheus, Grafana:** Система мониторинга. Сбор и визуализация метрик производительности и состояния системы.
    *   **Winston, Sentry:** Система логирования и отслеживания ошибок.

### 3. Архитектура Проекта: Взаимосвязь Компонентов Системы

Архитектура BrainMessenger следует принципам **модульности и разделения ответственности** (Принцип 9).

*   **Клиентские Приложения (Frontend):**
    *   Web (Next.js), Mobile & Desktop (React Native).
    *   Содержат пользовательский интерфейс и логику взаимодействия с пользователем.
    *   Взаимодействуют с Backend исключительно через **GraphQL API**.
    *   Могут напрямую скачивать файлы из Cloudflare R2 (если доступ публичный) или через Backend (если требуется аутентификация или подпись URL).
*   **API Gateway (В будущем):** Единая точка входа для всех клиентских запросов. Может выполнять аутентификацию, Rate Limiting, маршрутизацию на нужные Backend-сервисы. На старте роль Gateway выполняет сам Backend.
*   **Backend Service (NestJS):**
    *   Основной серверный компонент.
    *   Содержит **бизнес-логику** BrainMessenger (управление пользователями, чатами, сообщениями, звонками, платежами).
    *   Предоставляет **GraphQL API** для клиентских приложений.
    *   Взаимодействует с **Базой Данных (Neon/PostgreSQL через Prisma)** для чтения/записи структурированных данных.
    *   Взаимодействует с **Хранилищем Файлов (Cloudflare R2)** для загрузки/получения файлов (через AWS SDK).
    *   Взаимодействует с **Кэшем (Redis)** для быстрых операций.
    *   Ставит задачи в **Очередь (Kafka)** для асинхронной обработки.
    *   Использует внешние сервисы (Firebase, Stripe, Gmail API).
*   **База Данных (Neon/PostgreSQL):** Хранит структурированные данные. Масштабируется независимо.
*   **Хранилище Файлов (Cloudflare R2):** Хранит неструктурированные бинарные данные. Масштабируется по объему.
*   **Кэширование (Redis):** Быстрый доступ к временным данным.
*   **Очередь Сообщений (Kafka):** Буферизация и надежная доставка сообщений между сервисами для асинхронной обработки.
*   **Воркеры (Kafka Consumers):** Отдельные сервисы, читающие задачи из Kafka и выполняющие тяжелые или долгие операции (обработка изображений, отправка больших рассылок). Масштабируются независимо.
*   **Внешние Сервисы (Firebase, Stripe, Gmail API):** Предоставляют специализированный функционал (уведомления, платежи, email). Используются Backend-сервисом.

### 4. API: Интерфейс Взаимодействия

API BrainMessenger реализовано на **GraphQL** с использованием NestJS.
*   **Endpoint:** `/graphql`
*   **Структура:** Определяется GraphQL Схемой (типы User, Chat, Message и др.) и набором Queries (для получения данных) и Mutations (для изменения данных).
*   **Аутентификация:** JWT-токены в заголовке `Authorization: Bearer <token>`.
*   **Внешние API Интеграции:** Взаимодействие с внешними сервисами (Gmail, Stripe) осуществляется через Backend, который предоставляет соответствующие Mutations в GraphQL API (см. Спецификацию API).

**Ключевые функциональные области API:**
*   Аутентификация (регистрация, логин, логаут, 2FA).
*   Управление пользователями (получение/обновление/удаление профиля).
*   Управление чатами (создание, получение списка, получение данных чата).
*   Работа с сообщениями (отправка текстовых сообщений, отправка файлов, получение истории).
*   Управление контактами.
*   Настройки приложения (тема, уведомления, язык).
*   Управление Premium-подпиской.
*   Управление файлами (загрузка в R2 через Backend).
*   (В будущем) Звонки и видеозвонки (управление сессиями звонков).

**Подробная спецификация API описана в DocSpec.md.**

### 5. База Данных (Neon/PostgreSQL): Хранилище Основных Данных

*   **Технология:** PostgreSQL 15.x.
*   **Хостинг:** Neon.
*   **Роль:** Реляционная база данных для всех структурированных данных.
*   **Ключевые таблицы (определены в `backend/prisma/schema.prisma`):**
    *   `User`: Информация о пользователях (`id`, `email`, `password_hash`, `name`, `avatarUrl`, `createdAt`, `updatedAt`, `is2FaEnabled`).
    *   `Chat`: Информация о чатах (`id`, `type`, `name`, `createdAt`, `updatedAt`, `lastMessageAt`).
    *   `UserChat`: Связующая таблица для Many-to-Many связи между `User` и `Chat` (`userId`, `chatId`, `joinedAt`, `lastReadMessageId`).
    *   `Message`: Сообщения в чатах (`id`, `chatId`, `senderId`, `content`, `fileUrl`, `fileMetadata` (JSON), `createdAt`, `updatedAt`).
    *   `Contact`: Список контактов пользователя (`id`, `userId`, `contactId`).
    *   `Transaction`: История платежей (для Premium) (`id`, `userId`, `amount`, `currency`, `status`, `provider`, `createdAt`).
    *   `Code`: Временные коды (для email верификации, 2FA) (`id`, `userId/email`, `code`, `type`, `expiresAt`).
*   **Взаимодействие:** Только Backend через Prisma ORM.
*   **Оптимизация и Масштабирование:** Используются индексы, оптимизация запросов Prisma, материализованные представления, партиционирование, репликация (см. DocOptimizationIn).

### 6. Хранилище Файлов (Cloudflare R2): Хранилище Медиа и Бинарных Активов

*   **Технология:** Объектное хранилище, S3-совместимый API.
*   **Хостинг:** Cloudflare R2.
*   **Роль:** Хранение всех неструктурированных данных (файлы пользователей, аватарки, записи звонков).
*   **Взаимодействие:** Backend загружает файлы, Backend или Frontend получает файлы (напрямую или через подписанные URL).
*   **Структура:** Файлы организованы по папкам/ключам (например, `avatars/`, `chat-images/`, `sensitive-data/`).
*   **Безопасность:** Шифрование в покое, ограниченный доступ, шифрование чувствительных данных на уровне приложения перед загрузкой (см. DocInt, DocSecurity).

### 7. Проектная Структура (Монорепозиторий): Организация Кода

Проект организован как монорепозиторий с использованием Turborepo.
*   **Цель:** Упростить управление несколькими приложениями/пакетами (backend, web, mobile-desktop, core) и переиспользование кода.
*   **Пакеты:**
    *   `core`: Общий код.
    *   `backend`: Серверная логика.
    *   `web`: Веб-приложение.
    *   `mobile-desktop`: Мобильное/десктопное приложение (React Native).
*   **Инфраструктура:** Файлы конфигурации для развертывания (`infrastructure/`).
*   **Документация:** Проектная документация (`docs/`).
*   Подробное описание структуры монорепозитория см. в DocDevIn.md (раздел 2).

### 8. Безопасность: Фундаментальный Аспект Системы

Безопасность встроена на всех уровнях (см. подробнее в Руководстве по безопасности BrainMessenger).
*   **Аутентификация:** JWT, 2FA через email.
*   **Авторизация:** Проверка прав доступа к ресурсам на Backend.
*   **Шифрование:** TLS/SSL для передачи данных, шифрование паролей (bcrypt/argon2), шифрование чувствительных данных на уровне приложения перед хранением в R2.
*   **Валидация:** Строгая валидация входных данных на Backend.
*   **Защита от Атак:** Rate Limiting, безопасные HTTP-заголовки.
*   **Принцип:** Безопасность — это постоянный процесс **обучения** (Принцип 1) и **совершенствования** (Принцип 9), основанный на **ответственности** (Принцип 6).

### 9. Развертывание: Доставка Системы Пользователям

Процесс развертывания автоматизирован.
*   **Контейнеризация:** Docker-образы для Backend, Kafka Consumers.
*   **Оркестрация:** Kubernetes управляет контейнерами в облаке.
*   **Инфраструктура как Код:** Terraform для управления облачными ресурсами.
*   **CI/CD:** GitHub Actions автоматизирует сборку, тестирование и деплой.
*   Веб-приложение может деплоиться отдельно (например, на Vercel или Cloudflare Pages).
*   Подробное описание процесса деплоя см. в DocDevIn.md (раздел 9) и Руководстве по развертыванию.

### 10. Мониторинг и Логирование: Видимость Состояния Системы

*   **Мониторинг:** Сбор и визуализация метрик производительности и здоровья системы (CPU, RAM, задержки API/БД, ошибки). Инструменты: Prometheus, Grafana. Позволяет выявлять проблемы на ранней стадии (Принцип 9: Кайдзен).
*   **Логирование:** Сбор структурированных логов от всех компонентов (Backend, воркеры). Инструменты: Winston (для Node.js), Sentry (для отслеживания ошибок). Позволяет анализировать работу системы и диагностировать проблемы.
*   Подробное описание см. в Руководстве по оптимизации BrainMessenger (раздел 14) и Руководстве по мониторингу.

### 11. Внешние API Интеграции: Использование Внешних Рычагов

Backend взаимодействует с несколькими внешними сервисами через их API.
*   **Gmail API:** Отправка транзакционных email (подтверждение, 2FA).
*   **Stripe:** Обработка платежей и подписок.
*   **Firebase FCM:** Отправка push-уведомлений на мобильные устройства.
*   Использование этих сервисов — **прагматичный** (Принцип 12) способ быстро добавить функционал, используя готовые, **надежные** (Принцип 3) решения.
*   Подробное описание интеграций см. в DocInt.md.
