# Спецификация API BrainMessenger (Интерфейс Системы)

### 1. Общая Информация: API как Точка Взаимодействия с Системой

**Название проекта:** BrainMessenger

**Описание:** Этот документ описывает GraphQL API BrainMessenger — **основной интерфейс** (Принцип 9), через который клиентские приложения (Web, Mobile, Desktop) взаимодействуют с серверной частью системы и получают доступ к данным и функциональности. GraphQL выбран за его **эффективность и гибкость** (Принцип 9, 2), позволяя клиентам запрашивать только те данные, которые им необходимы.

**Цель:** Предоставить четкое и полное описание доступных запросов (Queries) и мутаций (Mutations), их входных и выходных данных, а также правил аутентификации и форматов ошибок. Это обеспечивает **системность** (Принцип 9) взаимодействия между Frontend и Backend и повышает **качество** (Принцип 3) разработки.

**Базовый URL:** `https://api.brainmessenger.com/graphql` (адрес API Gateway, который перенаправляет запросы на Backend Service).

**Формат запросов:** GraphQL-запросы (POST-запросы с `Content-Type: application/json` и телом в формате GraphQL).

**Аутентификация:** Для большинства методов требуется передать JWT-токен в заголовке `Authorization: Bearer <token>`. Токен получается после успешного логина. Это часть нашей **системы безопасности** (Принцип 3, 5).

**Связь с другими документами:**

*   DocDevIn.md: Детали реализации Backend на NestJS и GraphQL.

### 2. Схема GraphQL: Структура Доступных Данных и Операций

Полная схема GraphQL доступна по адресу API через инструменты интроспекции (например, GraphQL Playground, Apollo Studio). Ниже приведены основные типы и операции.

#### 2.1. Типы Данных (Schemas): Описание Структуры Информации

*   **User:** Представляет пользователя BrainMessenger.
    ```graphql
    type User {
      id: ID!
      email: String!
      name: String!
      avatarUrl: String # URL аватара пользователя (хранится в R2)
      createdAt: DateTime!
      updatedAt: DateTime!
      # ... другие поля (например, статус онлайн/оффлайн)
    }
    ```
*   **Chat:** Представляет чат (личный, групповой, канал).
    ```graphql
    type Chat {
      id: ID!
      name: String! # Название чата (для групп/каналов)
      type: String! # Тип чата ("personal", "group", "channel")
      users: [User!] # Список участников чата
      messages(pagination: PaginationInput): [Message!] # Сообщения в чате (с пагинацией)
      lastMessageAt: DateTime # Время последнего сообщения (для сортировки списка чатов)
      createdAt: DateTime!
      updatedAt: DateTime!
      # ... другие поля (например, фото группы/канала, описание)
    }
    ```
*   **Message:** Представляет сообщение в чате.
    ```graphql
    type Message {
      id: ID!
      content: String! # Текст сообщения (для текстовых сообщений)
      fileUrl: String # URL файла, если сообщение является файлом (хранится в R2)
      fileMetadata: JSON # Метаданные файла (тип, размер, имя) (хранится в Neon/R2)
      sender: User! # Отправитель сообщения
      chat: Chat! # Чат, которому принадлежит сообщение
      createdAt: DateTime! # Время отправки сообщения
      updatedAt: DateTime!
      # ... другие поля (например, статус прочтения, реакции)
    }
    ```
*   **PaginationInput:** Входной тип для пагинации списков.
    ```graphql
    input PaginationInput {
      take: Int # Максимальное количество элементов
      skip: Int # Пропустить элементов (для offset-based пагинации)
      cursor: String # Курсор (ID) для cursor-based пагинации
    }
    ```
*   **AuthPayload:** Тип, возвращаемый при успешной аутентификации.
    ```graphql
    type AuthPayload {
      token: String! # JWT токен доступа
      user: User! # Данные аутентифицированного пользователя
    }
    ```
*   **DateTime, JSON:** Стандартные скалярные типы или кастомные скаляры, определенные на Backend.

#### 2.2. Операции (Queries и Mutations): Доступ к Функционалу Системы

<h5>2.2.1. Аутентификация (Authentication)</h5>
*   **`mutation registerUser(input: RegisterInput!): AuthPayload!`**
    *   **Описание:** Регистрирует нового пользователя в системе.
    *   **`RegisterInput`:** `{ email: String!, password: String!, name: String! }` (password min 8 chars, digit, special char - см. валидацию).
    *   **Требуется аутентификация:** Нет.
    *   **Сценарий:** Пользователь заполняет форму регистрации.
    *   **Ошибки:** `400 Bad Request` (неверный формат данных/валидация), `409 Conflict` (email уже используется).

*   **`mutation loginUser(input: LoginInput!): AuthPayload!`**
    *   **Описание:** Аутентифицирует пользователя по email и паролю.
    *   **`LoginInput`:** `{ email: String!, password: String! }`.
    *   **Требуется аутентификация:** Нет.
    *   **Сценарий:** Пользователь вводит учетные данные для входа.
    *   **Ошибки:** `401 Unauthorized` (неверный email или пароль).

*   **`mutation logoutUser: Boolean!`**
    *   **Описание:** Инвалидирует текущий JWT токен на сервере (если реализовано хранение токенов или их черных списков).
    *   **Требуется аутентификация:** Да (используется текущий токен).
    *   **Сценарий:** Пользователь нажимает "Выйти".
    *   **Возвращает:** `true` при успешном выходе.
    *   **Ошибки:** `401 Unauthorized` (токен недействителен).

<h5>2.2.2. Пользователь (User)</h5>
*   **`query getUser(id: ID!): User`**
    *   **Описание:** Получает данные пользователя по его уникальному ID.
    *   **Требуется аутентификация:** Да (для доступа к любым данным пользователя, кроме, возможно, публичного профиля).
    *   **Сценарий:** Просмотр профиля пользователя.
    *   **Ошибки:** `404 Not Found` (пользователь с таким ID не найден).

*   **`query getCurrentUser: User!`**
    *   **Описание:** Получает данные текущего аутентифицированного пользователя.
    *   **Требуется аутентификация:** Да.
    *   **Сценарий:** Загрузка профиля текущего пользователя при старте приложения.
    *   **Ошибки:** `401 Unauthorized`.

*   **`mutation updateUser(id: ID!, input: UserInput!): User!`**
    *   **Описание:** Обновляет данные профиля пользователя.
    *   **`UserInput`:** `{ name: String, email: String, password: String, avatarUrl: String }`. Поля опциональны.
    *   **Требуется аутентификация:** Да (и пользователь должен быть владельцем профиля или иметь права администратора).
    *   **Сценарий:** Пользователь редактирует свой профиль.
    *   **Ошибки:** `400 Bad Request` (неверный формат данных), `401 Unauthorized`, `403 Forbidden` (нет прав на редактирование этого пользователя), `404 Not Found`.

*   **`mutation deleteUser(id: ID!): Boolean!`**
    *   **Описание:** Удаляет аккаунт пользователя.
    *   **Требуется аутентификация:** Да (и пользователь должен быть владельцем аккаунта).
    *   **Сценарий:** Пользователь решает удалить свою учетную запись.
    *   **Возвращает:** `true` при успешном удалении.
    *   **Ошибки:** `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.

<h5>2.2.3. Чаты (Chats)</h5>
*   **`query getChats: [Chat!]!`**
    *   **Описание:** Получает список всех чатов, в которых участвует текущий пользователь.
    *   **Требуется аутентификация:** Да.
    *   **Сценарий:** Загрузка списка чатов на главном экране.
    *   **Ошибки:** `401 Unauthorized`.

*   **`query getChat(id: ID!): Chat`**
    *   **Описание:** Получает данные конкретного чата по ID.
    *   **Требуется аутентификация:** Да (и пользователь должен быть участником этого чата).
    *   **Сценарий:** Открытие конкретного чата.
    *   **Ошибки:** `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.

*   **`mutation createChat(input: CreateChatInput!): Chat!`**
    *   **Описание:** Создает новый чат.
    *   **`CreateChatInput`:** `{ type: String!, name: String, userIds: [ID!]! }`. `type` может быть "personal", "group", "channel". `name` обязателен для "group" и "channel". `userIds` включает ID создателя и других участников (для "personal" и "group").
    *   **Требуется аутентификация:** Да.
    *   **Сценарий:** Пользователь создает новый чат или группу.
    *   **Ошибки:** `400 Bad Request` (неверный тип, отсутствуют userIds и т.п.), `401 Unauthorized`, `403 Forbidden` (нет прав на создание такого типа чата).

*   **`query getMessages(chatId: ID!, pagination: PaginationInput): [Message!]!`**
    *   **Описание:** Получает список сообщений для заданного чата. Поддерживает пагинацию.
    *   **Требуется аутентификация:** Да (и пользователь должен быть участником чата).
    *   **Сценарий:** Открытие чата, подгрузка старых сообщений при прокрутке.
    *   **Ошибки:** `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.

*   **`mutation sendMessage(chatId: ID!, content: String!, fileUrl: String, fileMetadata: JSON): Message!`**
    *   **Описание:** Отправляет новое сообщение в чат. Может содержать текст или ссылку на файл.
    *   **`chatId`:** ID чата. `content`: Текст сообщения. `fileUrl`: URL файла в R2 (опционально). `fileMetadata`: Метаданные файла (опционально).
    *   **Требуется аутентификация:** Да (и пользователь должен быть участником чата).
    *   **Сценарий:** Пользователь отправляет текстовое сообщение или файл.
    *   **Ошибки:** `400 Bad Request`, `401 Unauthorized`, `403 Forbidden` (нет прав писать в этот чат), `404 Not Found` (чат не найден).

*   **`mutation deleteMessage(messageId: ID!): Boolean!`**
    *   **Описание:** Удаляет сообщение.
    *   **Требуется аутентификация:** Да (и пользователь должен быть отправителем сообщения или иметь права администратора чата).
    *   **Сценарий:** Пользователь удаляет свое сообщение.
    *   **Возвращает:** `true` при успешном удалении.
    *   **Ошибки:** `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.

<h5>2.2.4. Внешние Интеграции (через API Backend)</h5>
*   **`mutation sendVerificationCode(email: String!): Boolean!`**
    *   **Описание:** Запускает процесс отправки кода подтверждения на указанный email (используя Gmail API). Используется при регистрации или сбросе пароля.
    *   **Требуется аутентификация:** Нет.
    *   **Сценарий:** Пользователь вводит email на экране регистрации/восстановления пароля.
    *   **Возвращает:** `true` при успешной постановке задачи на отправку.
    *   **Ошибки:** `400 Bad Request` (неверный формат email), `429 Too Many Requests` (защита от спама).

*   **`mutation verifyEmailCode(email: String!, code: String!): Boolean!`**
    *   **Описание:** Проверяет код подтверждения, полученный по email.
    *   **Требуется аутентификация:** Нет.
    *   **Сценарий:** Пользователь вводит код из email.
    *   **Возвращает:** `true` при успешной проверке.
    *   **Ошибки:** `400 Bad Request` (неверный код или email), `404 Not Found` (для данного email нет активного кода).

*   **`mutation enableTwoFactorAuth(userId: ID!): Boolean!`**
*   **`mutation disableTwoFactorAuth(userId: ID!): Boolean!`**
    *   **Описание:** Включает/отключает двухфакторную аутентификацию для пользователя. Требует предварительной верификации email. Код для 2FA отправляется на email (Gmail API) при каждом логине после включения.
    *   **Требуется аутентификация:** Да (пользователь должен управлять своим аккаунтом).
    *   **Сценарий:** Пользователь настраивает безопасность аккаунта.
    *   **Ошибки:** `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `400 Bad Request` (email не верифицирован).

*   **`mutation sendTwoFactorCode(userId: ID!): Boolean!`**
    *   **Описание:** Отправляет код двухфакторной аутентификации на email пользователя.
    *   **Требуется аутентификация:** Частичная (например, после ввода пароля, но до ввода 2FA кода). Логика зависит от флоу аутентификации.
    *   **Сценарий:** Пользователь входит в аккаунт с включенной 2FA.
    *   **Ошибки:** `404 Not Found` (пользователь не найден или 2FA не включена), `429 Too Many Requests`.

*   **`mutation createPaymentSession(input: PaymentInput!): PaymentSessionPayload!`**
    *   **Описание:** Создает сессию оплаты через Stripe для оформления Premium подписки.
    *   **`PaymentInput`:** `{ planId: ID!, returnUrl: String! }` (ID выбранного тарифного плана, URL для редиректа после оплаты).
    *   **`PaymentSessionPayload`:** `{ sessionId: String!, redirectUrl: String! }` (ID сессии и URL для перенаправления пользователя на страницу оплаты Stripe).
    *   **Требуется аутентификация:** Да.
    *   **Сценарий:** Пользователь нажимает "Подписаться" на Premium экране.
    *   **Ошибки:** `400 Bad Request` (неверный planId), `401 Unauthorized`.

### 3. Обработка Ошибок API: Ясная Обратная Связь (Принцип 3, 14)

API возвращает стандартизированные ошибки для **ясной обратной связи** (Принцип 14) с клиентом и поддержания **качества** (Принцип 3).

*   **Формат ошибок (в соответствии со спецификацией GraphQL):**
    При возникновении ошибки (валидации, аутентификации, бизнес-логики и т.д.) ответ GraphQL будет содержать поле `errors`:
    ```json
    {
      "data": null, // Обычно null при ошибках верхнего уровня
      "errors": [
        {
          "message": "Описание ошибки (может быть локализовано - см. DocLocIn)",
          "locations": [...], // Расположение ошибки в запросе
          "path": [...],      // Путь к полю, вызвавшему ошибку
          "extensions": {
            "code": "КОД_ОШИБКИ", // Стандартизированный код ошибки (например, "UNAUTHENTICATED", "FORBIDDEN", "BAD_USER_INPUT", "NOT_FOUND")
            "http": { // Если применимо, HTTP статус код
              "status": 401,
              "headers": {}
            },
             "details": {} // Дополнительные детали об ошибке (опционально)
          }
        }
        // ... могут быть другие ошибки
      ]
    }
    ```
*   **Ключевые коды ошибок (extensions.code):**
    *   `UNAUTHENTICATED`: Требуется аутентификация, но токен отсутствует или недействителен (`HTTP 401`).
    *   `FORBIDDEN`: Аутентификация пройдена, но у пользователя нет прав на выполнение операции (`HTTP 403`).
    *   `BAD_USER_INPUT`: Ошибка валидации входных данных (`HTTP 400`). Детали могут быть в `extensions.details`.
    *   `NOT_FOUND`: Запрошенный ресурс не найден (`HTTP 404`).
    *   `CONFLICT`: Конфликт данных (например, попытка зарегистрировать email, который уже существует) (`HTTP 409`).
    *   `INTERNAL_SERVER_ERROR`: Неожиданная ошибка на сервере (`HTTP 500`).
    *   `RATE_LIMITED`: Превышен лимит запросов (`HTTP 429`).
*   **Локализация ошибок:** Сообщения об ошибках (`message`) могут быть локализованы на стороне Backend на основе заголовка `Accept-Language` или предпочтений пользователя (см. DocLocIn).

### 4. Примечания

*   **Типы данных:**
    *   `ID`: Уникальный идентификатор ресурса (обычно строка).
    *   `String`: Текстовая строка.
    *   `Int`, `Float`: Числа.
    *   `Boolean`: Логическое значение (`true`/`false`).
    *   `DateTime`: Дата и время в формате ISO 8601 (например, "2025-03-14T10:30:00Z").
    *   `JSON`: Кастомный скаляр для передачи произвольных JSON-объектов (например, для `fileMetadata`).
    *   `!` в конце типа означает, что поле **не может быть null**.
*   **Пагинация:** Для списков, которые могут быть длинными (`messages`), используйте пагинацию для эффективной загрузки данных (Принцип 9).
*   **Валидация:** Backend выполняет строгую валидацию всех входных данных (через DTO и `class-validator`), возвращая ошибки `BAD_USER_INPUT` при их некорректности. Это часть **системы обеспечения качества** (Принцип 3).
*   **Безопасность:** Все запросы, работающие с чувствительными данными или изменяющие состояние системы (кроме регистрации/логина), требуют аутентификации.
*   **Инструменты тестирования API:** Используйте GraphQL Playground (часто доступен по адресу `/graphql` в dev-режиме Backend), Postman или Apollo Studio для отправки запросов и тестирования API.

```
